# 선수지식 부트캠프 이론 정리

> 앞으로 나올 백엔드 실습을 따라가기 전에, 요청과 응답이 어떻게 오가고 Git과 DB를 왜 쓰는지 먼저 감을 잡는 문서입니다.

> 이번 시퀀스 한 줄 요약  
> Postman으로 요청을 보내고, JSON을 읽고, Git 브랜치를 만들고, DB 표 구조를 말로 설명할 수 있게 만드는 준비 단계입니다.

## 먼저 이것만 기억해도 됩니다

- GET은 주로 조회, POST는 주로 생성에 사용합니다.
- JSON은 서버와 클라이언트가 데이터를 주고받을 때 자주 보는 형태입니다.
- branch는 내 작업을 다른 사람 작업과 섞지 않기 위해 나눕니다.

## 왜 이 시퀀스가 필요한가

이후 스프링 실습에서는 요청을 보내고 응답을 읽는 일이 계속 반복됩니다.
그런데 HTTP 메서드, 상태 코드, JSON 형태가 낯설면 코드보다 화면에서 먼저 막히기 쉽습니다.

Git도 마찬가지입니다.
브랜치를 왜 나누는지 모르면 실습을 시작하기도 전에 작업 흐름이 흐려집니다.

DB 기초도 먼저 알아두면 좋습니다.
나중에 저장 기능을 배우기 시작하면, 결국 무엇을 표 형태로 저장하는지 바로 연결해서 이해할 수 있기 때문입니다.

## 오늘 실습 흐름 한눈에 보기

1. 공개 연습 API에 GET 요청을 보냅니다.
2. 같은 API에 POST 요청을 보내고 응답을 확인합니다.
3. JSON body 값을 바꿔보며 요청 내용이 어떻게 달라지는지 봅니다.
4. 실습 레포를 clone 하고 새 branch를 만듭니다.
5. 변경 파일을 add 하고 commit 합니다.
6. 간단한 회원 테이블 예시로 table, row, column, PK를 정리합니다.

짧게 말하면 오늘은  
**요청 보내기 -> 응답 읽기 -> 값 바꾸기 -> 작업 분리하기 -> 데이터 표 이해하기** 흐름을 잡는 시간입니다.

## 핵심 개념 쉬운 설명

### HTTP 메서드

HTTP 메서드는 서버에게 "이번 요청으로 무엇을 하고 싶은지" 알려주는 말이라고 생각하시면 됩니다.

- `GET`: 데이터를 가져오고 싶을 때 많이 씁니다.
- `POST`: 새로운 데이터를 보내서 만들고 싶을 때 많이 씁니다.

이번 실습에서는 Postman으로 두 요청을 직접 보내보며 차이를 확인합니다.
나중에 스프링 실습에서 `@GetMapping`, `@PostMapping`을 볼 때도 결국 이 감각이 그대로 이어집니다.

### 상태 코드

상태 코드는 서버가 이번 요청을 어떻게 처리했는지 짧게 알려주는 숫자입니다.

- `200 OK`: 요청이 잘 처리되었습니다.
- `201 Created`: 새 데이터가 만들어졌습니다.
- `400 Bad Request`: 요청 형식이 잘못되었습니다.
- `404 Not Found`: 찾으려는 대상이 없습니다.

이번 실습에서는 응답 본문만 보지 말고 상태 코드도 함께 확인하는 습관을 잡는 것이 중요합니다.

### JSON 요청/응답

JSON은 데이터를 키와 값으로 정리해서 보여주는 형식입니다.
백엔드 실습에서는 요청 body와 응답 body에서 아주 자주 보게 됩니다.

예를 들어 아래처럼 볼 수 있습니다.

```json
{
  "title": "A&I Bootcamp",
  "body": "JSON 값을 수정해봅니다.",
  "userId": 4
}
```

여기서 중요한 것은 문법을 길게 외우는 것이 아닙니다.
어떤 키가 있고, 어떤 값을 바꾸면 요청 의미가 바뀌는지를 보는 것이 먼저입니다.

### Git branch

branch는 같은 레포 안에서 작업 줄기를 따로 나누는 도구입니다.
쉽게 말하면 "내 작업 공간을 잠깐 따로 빼서 안전하게 작업하는 것"에 가깝습니다.

실습에서 branch가 필요한 이유는 분명합니다.

- 기본 브랜치를 지키기 쉽습니다.
- 각자 작업 내용을 따로 남길 수 있습니다.
- 나중에 비교하거나 합칠 때 흐름이 선명해집니다.

이번 실습에서는 `git checkout -b feat/<이름>` 형태를 따라 해보며 감각을 잡습니다.

### DB의 table / row / column / PK

DB는 데이터를 표처럼 저장한다고 생각하면 이해가 쉬워집니다.

- `table`: 데이터를 담는 큰 표입니다.
- `row`: 표의 한 줄 데이터입니다.
- `column`: 각 칸의 종류입니다.
- `PK`: 각 row를 겹치지 않게 구분하는 대표 값입니다.

예를 들어 회원 표가 있다고 생각해보세요.

| id | name | email |
| --- | --- | --- |
| 1 | Mina | mina@aandi.dev |
| 2 | Jisoo | jisoo@aandi.dev |

여기서 `members`가 table이고, `1 / Mina / mina@aandi.dev` 한 줄이 row입니다.
`id`, `name`, `email`은 column이고, `id`는 회원을 구분하는 PK 역할을 합니다.

이 감각이 있으면 나중에 저장 구조를 배우기 시작할 때도 "결국 이 표와 연결되는구나" 하고 더 쉽게 이어집니다.

## 자주 헷갈리는 포인트

- GET과 POST는 URL만 다른 것이 아니라 요청 의도가 다릅니다.
- 응답 본문이 보여도 상태 코드가 실패일 수 있으니 같이 봐야 합니다.
- branch는 폴더를 새로 만드는 것이 아니라 작업 흐름을 나누는 것입니다.
- PK는 "중요한 값"이 아니라 "겹치지 않게 구분하는 값"에 더 가깝습니다.

## 오늘 꼭 기억할 것

오늘은 코드를 많이 만드는 날이 아닙니다.
대신 이후 실습에서 계속 반복될 기본 장면을 먼저 익히는 날입니다.

- 요청을 보낼 수 있어야 합니다.
- 응답을 읽을 수 있어야 합니다.
- 작업을 branch로 나눌 수 있어야 합니다.
- 데이터가 표 구조로 저장된다는 감각이 있어야 합니다.
