# 선수지식 부트캠프 이론 정리

이번 시퀀스는 이후 백엔드 실습을 따라가기 전에 요청, 응답, JSON, Git, DB 표 구조를 먼저 익히는 단계입니다.

## 먼저 이것만 기억해도 됩니다

- GET은 주로 조회에 사용합니다.
- POST는 주로 생성 요청에 사용합니다.
- JSON은 서버와 클라이언트가 데이터를 주고받을 때 자주 쓰는 형식입니다.
- branch는 내 작업을 다른 작업과 분리하기 위해 사용합니다.
- DB table은 데이터를 표처럼 저장하는 구조입니다.

## 왜 이 시퀀스가 필요한가

이후 Spring Boot 실습에서는 요청을 보내고 응답을 읽는 일이 계속 반복됩니다.
HTTP 메서드, 상태 코드, JSON 형태가 낯설면 코드를 보기 전에 API 도구에서 먼저 막히기 쉽습니다.

Git도 마찬가지입니다.
브랜치를 왜 나누는지 모르면 실습을 시작하기 전에 작업 흐름이 흐려집니다.

DB 기초도 먼저 알아두면 좋습니다.
나중에 저장 기능을 배우기 시작하면, 결국 무엇을 표 형태로 저장하는지 바로 연결해서 이해할 수 있기 때문입니다.

실무에서도 API가 안 맞을 때는 코드보다 요청, 응답, 상태 코드, 저장 형태를 먼저 확인하는 경우가 많습니다.

## 기초 개념

### HTTP 메서드

HTTP 메서드는 서버에게 이번 요청으로 무엇을 하고 싶은지 알려주는 말입니다.

- `GET`: 데이터를 가져오고 싶을 때 많이 씁니다.
- `POST`: 새로운 데이터를 보내서 만들고 싶을 때 많이 씁니다.

### 상태 코드

상태 코드는 서버가 이번 요청을 어떻게 처리했는지 알려주는 숫자입니다.

- `200 OK`: 요청이 잘 처리되었습니다.
- `201 Created`: 새 데이터가 만들어졌습니다.
- `400 Bad Request`: 요청 형식이 잘못되었습니다.
- `404 Not Found`: 찾으려는 대상이 없습니다.

### JSON

JSON은 데이터를 key와 value로 정리해서 보여주는 형식입니다.

```json
{
  "title": "A&I Bootcamp",
  "body": "JSON 값을 수정해봅니다.",
  "userId": 4
}
```

처음에는 문법을 길게 외우기보다 어떤 key가 있고 어떤 value를 바꾸면 요청 의미가 달라지는지 보는 것이 중요합니다.

### Git branch

branch는 같은 레포 안에서 작업 줄기를 따로 나누는 도구입니다.
실습에서는 기본 브랜치를 바로 건드리지 않고 내 작업을 따로 남기는 감각을 잡는 것이 중요합니다.

### DB table, row, column, PK

DB는 데이터를 표처럼 저장한다고 생각하면 이해하기 쉽습니다.

| id | name | email |
| --- | --- | --- |
| 1 | Mina | mina@aandi.dev |
| 2 | Jisoo | jisoo@aandi.dev |

- table: 데이터를 담는 큰 표입니다.
- row: 표의 한 줄 데이터입니다.
- column: 각 칸의 종류입니다.
- PK: 각 row를 겹치지 않게 구분하는 대표 값입니다.

## 현재 자료 흐름

이번 시퀀스는 코드 구현보다 손동작이 중요합니다.

1. `starter/http/get-post.http`로 GET 요청 모양을 봅니다.
2. `starter/http/create-post.http`로 POST 요청 모양을 봅니다.
3. `starter/json/create-post-request.json`에서 body 값을 바꿉니다.
4. `starter/git/command-flow.txt`로 Git 명령 흐름을 따라갑니다.
5. `starter/db/members-table-diagram.txt`로 DB 표 구조를 설명합니다.

## 자주 헷갈리는 포인트

- GET과 POST는 URL만 다른 것이 아니라 요청 의도가 다릅니다.
- 응답 body가 보여도 상태 코드가 실패일 수 있으니 같이 봐야 합니다.
- branch는 폴더를 새로 만드는 것이 아니라 작업 흐름을 나누는 것입니다.
- PK는 중요한 값이라는 뜻보다 겹치지 않게 구분하는 값에 가깝습니다.
